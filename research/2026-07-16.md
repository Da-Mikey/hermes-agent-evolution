# Research Report - 2026-07-16

## New Features

### [FEATURE] Agentic Harness Engineering with Observability-Driven Evolution
- **Source**: https://arxiv.org/html/2604.25850v3 (Agentic Harness Engineering: Observability-Driven Automatic Evolution of Coding-Agent Harnesses, Fudan University & Shanghai Qiji Zhifeng)
- **Impact**: High
- **Effort**: Medium
- **Priority Score**: 2.4 (0.9 * 4 / 1.5)

The AHE paper introduces a rigorous closed-loop for automatic harness evolution built on three observability pillars: component observability (every editable harness component gets a file-level representation for clean revertible edits), experience observability (trajectories distilled into structured root causes rather than raw logs), and decision observability (each edit ships with a self-declared prediction about which tasks it fixes and which it might regress, verified next round). This directly mirrors Hermes' existing Hephaestus forge pipeline but adds the key missing piece: falsifiable edit predictions with automated regression detection. AHE lifts pass@1 on Terminal-Bench 2 from 69.7% to 77.0%. Hermes already has the nightly forge cycle — what's missing is self-attribution per edit. Adding a "change manifest" that records each edit's predicted impact and checks it against next-cycle deltas would close the loop completely, allowing automatic reversion of edits that don't deliver or that regress.

### [FEATURE] Structured Memory Architecture (Four-Type Agent Memory)
- **Source**: https://blogs.oracle.com/developers/oracle-ai-agent-memory-a-governed-unified-memory-core-for-enterprise-ai-agents (Oracle AI Agent Memory, May 2026)
- **Impact**: High
- **Effort**: Medium
- **Priority Score**: 2.0 (0.8 * 5 / 2.0)

Oracle released a governed, unified memory substrate for AI agents that formalises four memory types: working (active conversation state), semantic (durable facts and preferences), episodic (past experiences and sessions), and procedural (behavioral rules and tool preferences). It scores 93.8% on LongMemEval with gpt-5.5 and holds per-turn cost flat at ~1,300 tokens (vs 13,900 for flat-history baseline over 80 turns). The key insight is that these four access patterns should share one substrate rather than four stitched-together services. Hermes currently uses a simpler session-based memory — adding a structured four-type memory plane with automatic summarisation triggers and per-user scoped retrieval would dramatically improve long-horizon task performance and is exactly the kind of substrate that makes the Oracular Dream more effective.

### [FEATURE] Self-Evolving Agent Framework with Experience-Driven Lifelong Learning
- **Source**: https://arxiv.org/abs/2508.19005 (Building Self-Evolving Agents via Experience-Driven Lifelong Learning: A Framework and Benchmark, ECNU)
- **Impact**: Medium
- **Effort**: High
- **Priority Score**: 1.3 (0.7 * 3 / 1.5)

This paper introduces a framework and benchmark (StuLife) for agents that accumulate experience across tasks and use it to self-improve. The framework defines experience-driven lifelong learning as a separate class of agent evolution — distinct from single-task optimisation — where the agent must decide what to retain, generalise, and forget. Hermes already has experience accumulation across sessions (The Loom, Oracular Dream) but lacks explicit mechanisms for selective forgetting or cross-task generalisation. The StuLife benchmark provides a standardised evaluation protocol for measuring whether an agent's experience-driven improvements actually transfer. This is a longer-term research integration rather than an immediate feature.

## Improvements

### [IMPROVEMENT] Change Manifest with Self-Declared Predictions for Forge Edits
- **Source**: AHE paper (above) — https://arxiv.org/html/2604.25850v3
- **Impact**: High
- **Effort**: Low-Medium
- **Priority Score**: 2.0 (0.9 * 3 / 1.0)

The highest-impact actionable finding. AHE's third observability pillar (decision observability) requires each harness edit to include a self-declared prediction: "this edit will fix tasks X, Y and might regress task Z." The prediction is verified against the next evaluation round. Edits whose predictions fail are auto-reverted at file granularity. Hermes' Hephaestus forge currently evolves the agent but lacks this attribution mechanism — edits are applied, evaluated in aggregate, but not tracked per-edit with falsifiable contracts. Implementing a lightweight change manifest (a structured YAML or JSON file per forge cycle recording each edit, its predicted deltas, and next-cycle verification status) would dramatically improve the forge's signal-to-noise ratio. Estimated ~50-100 LOC.

### [IMPROVEMENT] Prompt Cost Bounding Through Automatic Summarisation Triggers
- **Source**: Oracle AI Agent Memory (above) — https://blogs.oracle.com/developers/oracle-ai-agent-memory-a-governed-unified-memory-core-for-enterprise-ai-agents
- **Impact**: Medium
- **Effort**: Low
- **Priority Score**: 1.6 (0.8 * 2 / 1.0)

Oracle showed that a configurable summarisation trigger (tunable from 10k to 70k tokens) keeps per-turn cost bounded as conversations extend. In an 80-turn conversation, the memory-managed agent used ~1,300 tokens/turn vs 13,900 for flat history. Hermes' long-running sessions (research, analysis) would benefit from a simple automatic compaction trigger that summarises accumulated context when a threshold is crossed. The user already pays token costs — this reduces waste. Implementation is straightforward: monitor cumulative token count per session, trigger a summarisation pass when exceeding configurable threshold, maintain extracted durable memories separately.

### [IMPROVEMENT] Harness Evolution Evaluation Against Test-Time Scaling Baselines
- **Source**: https://arxiv.org/abs/2607.12227 (Rethinking the Evaluation of Harness Evolution for Agents, 2025)
- **Impact**: Medium
- **Effort**: Low
- **Priority Score**: 1.4 (0.7 * 2 / 1.0)

This paper revisits harness evolution evaluation and finds that naive test-time scaling (more sampling, more iterations) can match or exceed complex evolved harnesses on many benchmarks. The critical insight: harness evolution evaluations should compare against test-time scaling baselines, not just static baselines. Hermes' forge evaluations currently compare evolved vs non-evolved performance — adding a test-time scaling baseline (e.g., same agent with 2x/4x sampling) would validate whether forge improvements are genuinely structural rather than just a function of "more thinking." Easy to implement: add a baseline run with increased max_tool_iterations or n_samples alongside the evolved-vs-static comparison.

### [IMPROVEMENT] ReasoningBank-Style Experience Accumulation for Cross-Session Learning
- **Source**: https://arxiv.org/abs/2509.25140 (ReasoningBank: Scaling Agent Self-Evolving with Reasoning Memory, 2025)
- **Impact**: Medium
- **Effort**: Medium
- **Priority Score**: 1.2 (0.6 * 2 / 1.0)

ReasoningBank proposes a dedicated "reasoning memory" that stores structured reasoning traces from past sessions and makes them retrievable for future tasks. This is distinct from general memory or tool-use history — it specifically stores the reasoning patterns that worked, including failure modes and recovery strategies. Hermes' Loom currently weaves patterns from experience, but this happens implicitly during the nightly dream cycle. An explicit, queryable ReasoningBank that Hermes can consult mid-task (not just during offline evolution) would allow real-time pattern reuse. The Oracular Dream already does this at the agent-behaviour level; a ReasoningBank makes it retrievable at inference time too.

## Replacements

### [REPLACEMENT] Migrate AutoGen-Style Multi-Agent Orchestration to Direct Subagent Delegation
- **Source**: https://github.com/microsoft/autogen (AutoGen is in maintenance mode since April 2026; redirected to Microsoft Agent Framework)
- **Impact**: Low
- **Effort**: Medium
- **Priority Score**: 0.8 (0.4 * 2 / 1.0)

AutoGen, the most starred multi-agent framework (59.8k ★), is officially in maintenance mode. Its successor is Microsoft Agent Framework (MAF), which introduces graph-based orchestration, A2A and MCP interoperability, and cross-language support (.NET + Python). Hermes already has a lightweight multi-agent system (Argus, The Moirai, etc.) and doesn't fundamentally depend on AutoGen patterns. No migration is needed — the replacement signal is simply that AutoGen is no longer a source of new patterns to track. The MAF direction (graph-based orchestration, cross-language, A2A/MCP) is worth monitoring but not urgent to implement.

### [REPLACEMENT] Claude Agent SDK Tool Loop Replacing Custom Tool Plumbing
- **Source**: https://www.totalum.app/blog/claude-agent-sdk-totalum-2026 (Claude Agent SDK overview, May 2026)
- **Impact**: Low
- **Effort**: Medium
- **Priority Score**: 0.7 (0.5 * 2 / 1.5)

Anthropic's Claude Agent SDK (formerly Claude Code SDK) has become the default way to wire Claude into production workflows. It exposes Anthropic's internal agent harness as a library with built-in MCP client, subagents, sessions, and human-in-the-loop checkpoints. The key development for Hermes: Anthropic introduced "Managed Agents" with a scheduler, dreaming pass, and rubric-based outcome grading on top of SDK primitives. This dreaming pass pattern (offline consolidation of experience across sessions) mirrors Hermes' Oracular Dream directly. The finding isn't that Hermes should replace its stack — it's validation that Hermes' architecture (offline dreaming, agent loop, MCP tool use) is now industry standard. The main action item is to ensure Hermes' MCP integration stays current with Anthropic's protocol updates.
